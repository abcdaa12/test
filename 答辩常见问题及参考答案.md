# 答辩常见问题及参考答案

> 项目名称：DormSync — 基于 UniApp 与 Node.js 的宿舍协同管理系统
>
> 本文档整理了毕业设计答辩中高频出现的问题及参考回答，供答辩前复习使用。

---

## 一、选题与背景类

### 1. 为什么选择这个题目？

大学宿舍是大学生日常生活的核心场景，室友之间存在大量需要协同处理的事务，比如值日排班、费用分摊、集体决策等。目前这些事务主要依赖微信群聊沟通，存在信息碎片化、容易遗漏、难以追溯等问题。我希望开发一个专门面向宿舍场景的协同管理工具，将这些高频事务数字化、结构化，提升室友之间的协作效率。同时，微信小程序具有"即用即走"的特点，非常适合这种轻量级、高频次的使用场景。

### 2. 这个系统要解决什么问题？

主要解决以下痛点：
- **信息分散**：费用、排班、投票等信息散落在微信群聊中，难以查找和追溯
- **沟通低效**：群聊中讨论容易被刷屏，重要信息被淹没
- **缺乏结构化管理**：费用分摊靠口头记账，排班靠手动排表，容易出错和遗忘
- **协作不透明**：谁交了钱、谁投了票、轮到谁值日，缺乏清晰的记录和展示

### 3. 国内外有没有类似的系统？你的系统和它们有什么区别？

国内外确实有一些相关产品：
- **Splitwise**：专注于费用分摊，但不支持排班、投票等宿舍场景功能
- **Doodle**：专注于日程投票，功能单一
- **微信群公告**：功能简单，无法进行结构化的事务管理
- **一些宿舍管理系统**：多面向学校管理层（宿管、辅导员），侧重于考勤、报修等行政管理

我的系统区别在于：**面向宿舍成员自身**，聚焦室友之间的日常协同需求，将费用管理、投票决策、排班管理、消息通知等功能整合在一个微信小程序中，做到开箱即用、无需下载安装。

### 4. 你的系统有哪些创新点？

1. **场景聚焦**：专门面向宿舍这一特定场景，功能设计贴合室友协作的真实需求，而非泛化的团队协作工具
2. **多维协同一体化**：将费用分摊、投票决策、排班管理、公告通知、待办事项等多个维度的协同功能整合在一个平台中
3. **AI 智能排班**：集成 DeepSeek 大模型 API，支持用户用自然语言描述排班需求（如"小明周三有课不排"），AI 自动生成合理排班表
4. **国际化支持**：实现了中文、英文、韩语、日语四种语言的动态切换，适应留学生混住的宿舍场景
5. **微信生态深度集成**：利用微信一键登录、分享卡片邀请入舍等能力，降低使用门槛
6. **实时消息驱动**：所有协同操作（发起费用、创建投票等）自动触发消息通知，确保信息不遗漏

### 5. 你的系统面向的用户群体是谁？

主要面向**在校大学生**，特别是同一宿舍的室友群体。也适用于合租公寓的年轻人。核心用户画像是：需要与室友频繁协调日常事务、习惯使用微信的年轻用户。

### 6. 你做过需求调研吗？怎么确定这些功能是用户需要的？

在开发前，我通过以下方式进行了需求调研：
- 与身边同学交流，了解宿舍日常管理中的痛点
- 观察微信群聊中室友之间的高频沟通内容（费用AA、值日安排、集体决策等）
- 参考现有产品（Splitwise、微信群公告等）的功能和不足
- 最终提炼出费用管理、投票决策、排班管理、消息通知四大核心功能模块

## 二、技术选型类

### 7. 为什么选择 UniApp 作为前端框架？

选择 UniApp 主要基于以下考虑：
- **跨平台能力**：一套代码可以编译到微信小程序、H5、App 等多个平台，开发效率高
- **Vue 生态**：UniApp 基于 Vue.js，我使用了 Vue 3 的 Composition API，代码组织更灵活
- **微信小程序适配好**：UniApp 对微信小程序的支持非常成熟，API 封装完善
- **开发工具链完善**：HBuilderX 提供了良好的开发、调试、编译体验

### 8. 为什么后端选择 Node.js + Express，而不是 Java/Spring Boot？

- **开发效率**：Node.js 配合 Express 框架，可以快速搭建 RESTful API，适合中小型项目的快速迭代
- **前后端语言统一**：前端使用 JavaScript（Vue），后端也使用 JavaScript（Node.js），降低了技术栈切换成本
- **轻量灵活**：Express 是一个极简的 Web 框架，中间件机制灵活，适合本项目的规模
- **生态丰富**：npm 生态提供了 JWT 认证（jsonwebtoken）、文件上传（multer）、数据库驱动（mongoose）等成熟的库
- **异步 I/O**：Node.js 的事件驱动和非阻塞 I/O 模型适合处理高并发的消息通知等场景

### 9. 为什么选择 MongoDB 而不是 MySQL？

- **数据结构灵活**：宿舍管理系统中，投票选项、排班条目等数据结构不固定，MongoDB 的文档模型天然适合存储这类嵌套、变长的数据。例如 Decision 模型中的 options 数组和 voters 数组，用关系型数据库需要额外建表，而 MongoDB 可以直接嵌入
- **开发效率高**：使用 Mongoose ODM，可以直接用 JavaScript 对象定义数据模型，与 Node.js 配合无缝
- **云数据库支持**：使用 MongoDB Atlas 云数据库，免去了自建数据库服务器的运维成本
- **JSON 原生支持**：前后端数据交互使用 JSON 格式，MongoDB 存储的就是 BSON（二进制 JSON），减少了数据格式转换

### 10. 为什么用 JWT 而不是 Session？JWT 是怎么实现的？

JWT（JSON Web Token）的实现流程：
1. 用户通过微信登录后，后端用微信 openid 生成一个 JWT 令牌（有效期 7 天）
2. 前端将令牌存储在本地（uni.setStorageSync）
3. 每次请求时，前端在请求头中携带 `Authorization: Bearer <token>`
4. 后端 authMiddleware 中间件验证令牌的有效性，解析出 userId 和 openid 挂载到 req 对象上

选择 JWT 而不是 Session 的原因：
- **无状态**：JWT 不需要在服务端存储会话信息，减轻服务器压力
- **适合小程序**：微信小程序不支持 Cookie，而 JWT 通过请求头传递，天然适配
- **跨域友好**：前后端分离架构下，JWT 不受跨域限制

### 11. 你用了哪些第三方库？分别起什么作用？

后端依赖：
| 库名 | 作用 |
|------|------|
| express | Web 框架，处理路由和中间件 |
| mongoose | MongoDB ODM，定义数据模型和操作数据库 |
| jsonwebtoken | 生成和验证 JWT 令牌 |
| axios | 调用微信登录接口和 DeepSeek AI 接口 |
| cors | 处理跨域请求 |
| dotenv | 加载 .env 环境变量 |
| multer | 处理文件上传（头像） |

前端主要使用 UniApp 内置 API（uni.request、uni.login 等），未引入额外 UI 框架，所有界面均为手写实现。

### 12. 为什么不用现成的 UI 框架（如 uView、Vant Weapp）？

主要考虑：
- **减少依赖**：第三方 UI 框架会增加包体积，微信小程序对包大小有限制（主包 2MB）
- **定制灵活**：手写样式可以完全控制界面风格，方便实现夜间模式等主题切换
- **学习目的**：毕设项目希望展示自己的前端能力，而不是简单调用组件库
- **项目规模适中**：页面数量不多，手写 CSS 的工作量可控

### 13. Mongoose 是什么？为什么要用它？

Mongoose 是 MongoDB 的 ODM（Object Document Mapping，对象文档映射）库，类似于关系型数据库中的 ORM。使用它的原因：
- **Schema 定义**：可以为 MongoDB 的文档定义结构化的 Schema，包括字段类型、默认值、校验规则（required、enum 等）
- **数据校验**：在数据写入前自动校验，防止非法数据入库
- **关联查询**：通过 populate 方法实现类似 SQL JOIN 的关联查询
- **中间件钩子**：支持 pre/post 钩子，可以在保存前后执行自定义逻辑

## 三、系统架构与设计类

### 14. 系统的整体架构是怎样的？

系统采用经典的**前后端分离 + MVC 架构**，分为三层：
- **表现层（前端）**：UniApp 开发的微信小程序，负责用户交互和界面展示
- **业务逻辑层（后端）**：Node.js + Express 构建的 RESTful API 服务，按 MVC 模式组织代码——routes 定义路由、controllers 处理业务逻辑、models 定义数据模型
- **数据持久层**：MongoDB Atlas 云数据库，存储所有业务数据

前端通过封装的 request.js 发起 HTTP 请求与后端通信，后端通过 Mongoose 操作数据库。中间件层（middlewares）负责跨域处理、JWT 认证、全局错误处理等横切关注点。

### 15. 你的后端代码是怎么分层的？为什么这样分？

后端采用 MVC 分层架构：
```
DormSync-server/
├── app.js              # 入口文件，组装中间件和路由
├── config/db.js        # 数据库连接配置
├── models/             # M - 数据模型层（Mongoose Schema）
├── controllers/        # C - 控制器层（业务逻辑）
├── routes/             # 路由层（URL 映射到控制器方法）
├── middlewares/         # 中间件层（JWT、CORS、错误处理）
└── uploads/            # 静态资源（头像文件）
```

这样分层的好处：
- **职责清晰**：每一层只关注自己的职责，降低耦合
- **易于维护**：修改业务逻辑只需改 controller，修改数据结构只需改 model
- **可复用**：中间件可以在多个路由间复用（如 authMiddleware）

### 16. 数据库设计了哪些集合？关键的关联关系是什么？

系统设计了 9 个数据集合：

| 集合 | 说明 | 关键字段 |
|------|------|----------|
| User | 用户信息 | openid, dormId(关联宿舍), role(角色) |
| Dorm | 宿舍信息 | dormNumber, members(成员数组), creator |
| Finance | 费用记录 | dormId, title, amount, creatorId, payeeIds, status |
| Decision | 投票决策 | dormId, options(选项数组), voters(投票记录), deadline |
| Schedule | 排班记录 | dormId, weekLabel, cycle, items(每日排班) |
| Task | 事务安排 | dormId, assigneeId, deadline, status |
| Todo | 待办事项 | userId, type, relatedId, status |
| Message | 消息通知 | userId, type, content, status |
| Announce | 宿舍公告 | dormId, title, content, readBy(已读列表) |

核心关联关系：
- User 与 Dorm 是**多对一**关系（多个用户属于一个宿舍）
- Dorm 与 Finance/Decision/Schedule/Announce 是**一对多**关系（一个宿舍有多条记录）
- User 与 Todo/Message 是**一对多**关系（一个用户有多条待办/消息）
- Decision 中嵌套了 voters 数组，记录每个用户的投票选择（**嵌入式文档设计**）

### 17. 为什么有些数据用嵌入式文档，有些用引用？

这是 MongoDB 数据建模中的经典取舍：
- **嵌入式文档**（如 Decision 中的 options 和 voters）：数据与父文档生命周期一致、查询时经常一起读取、数据量不会无限增长 → 适合嵌入
- **引用**（如 User 的 dormId 引用 Dorm）：数据独立存在、可能被多处引用、需要独立更新 → 适合引用

例如投票的选项和投票记录，它们只在投票详情中使用，不会被其他地方引用，所以嵌入到 Decision 文档中，查询时一次读取即可，避免了多次查询。

### 18. RESTful API 是怎么设计的？遵循了哪些规范？

API 设计遵循 RESTful 风格：
- **统一前缀**：所有接口以 `/api` 开头，按资源分组（/api/user、/api/dorm、/api/finance 等）
- **HTTP 方法语义化**：GET 查询、POST 创建、PUT 更新、DELETE 删除
- **统一响应格式**：所有接口返回 `{ code, msg, data }` 结构，code 200 表示成功，400 表示参数错误，401 表示未认证，404 表示资源不存在，500 表示服务器错误
- **无状态**：每次请求通过 JWT 令牌携带身份信息，服务端不存储会话状态

示例：
```
GET    /api/finance/list?dormId=xxx    → 查询费用列表
POST   /api/finance/create             → 创建费用记录
DELETE /api/finance/delete             → 删除费用记录
PUT    /api/message/update             → 更新消息状态
```

### 19. 前端的页面路由是怎么组织的？

使用 UniApp 的 pages.json 配置页面路由，共 17 个页面：
- **TabBar 页面**（4个）：首页、消息、更多、我的
- **功能页面**：费用管理、投票决策、排班管理、公告管理
- **用户相关**：登录、个人信息编辑、宿舍设置
- **宿舍管理**：邀请成员、宿舍成员、宿舍长选举
- **其他**：语言切换、关于我们、用户须知

登录页设为第一个页面，App.vue 启动时检查登录态，已登录则自动跳转首页。

### 20. 前端的请求封装是怎么做的？

在 `utils/request.js` 中基于 `uni.request` 进行了统一封装，实现了类似 axios 拦截器的功能：
- **请求拦截**：自动从本地存储读取 JWT 令牌，附加到请求头的 Authorization 字段；设置统一的 Content-Type 和超时时间（60秒）
- **响应拦截**：统一处理 HTTP 状态码，401 时自动清除登录态并跳转登录页，其他错误弹出 Toast 提示
- **便捷方法**：导出 get/post/put/del 四个快捷方法，调用时只需传入路径和参数
- **错误处理**：网络异常时统一弹出"网络异常，请稍后重试"提示

## 四、核心功能实现类

### 21. 微信登录的完整流程是怎样的？

1. 前端调用 `uni.login()` 获取临时登录凭证 code
2. 将 code 发送到后端 `/api/user/login` 接口
3. 后端使用 code + AppID + AppSecret 调用微信服务器的 `jscode2session` 接口，换取用户的 openid
4. 后端根据 openid 在 User 集合中查找用户，不存在则自动创建新用户（默认昵称"宿舍成员"）
5. 生成 JWT 令牌（载荷包含 userId 和 openid，有效期 7 天）返回给前端
6. 前端存储 token、userInfo、userId 到本地
7. 如果是新用户（昵称为默认值），跳转到个人信息编辑页；如果没有宿舍，跳转到宿舍设置页；否则进入首页

### 22. 宿舍的创建和加入流程是怎样的？

**创建宿舍**：
1. 用户输入宿舍号（如"5号楼306"）
2. 后端检查用户是否已有宿舍（防止重复创建）
3. 检查宿舍号是否已存在（防止重复）
4. 创建 Dorm 文档，将用户加入 members 数组，设为 creator
5. 更新用户的 dormId 和 role（设为 leader 宿舍长）

**加入宿舍**（三种方式）：
- **宿舍号加入**：手动输入宿舍号，后端查找匹配的宿舍并加入
- **分享卡片加入**：通过微信分享卡片，卡片携带 dormNumber 参数，接收者打开后自动预填宿舍号
- **手机号邀请**：宿舍长输入对方手机号，后端根据手机号查找用户并直接加入

### 23. 分享卡片邀请是怎么实现的？

利用微信小程序的 `onShareAppMessage` 能力：
1. 在首页定义分享回调，返回分享标题和路径：`/pages/login/login?dormNumber=xxx`
2. 被邀请者点击分享卡片后，小程序启动，App.vue 的 onLaunch 捕获 query 参数中的 dormNumber
3. 将 dormNumber 存入本地存储（inviteDormNumber）
4. 登录完成后，检测到 inviteDormNumber 存在且用户没有宿舍，自动跳转到宿舍设置页并预填宿舍号
5. 用户确认加入后清除 inviteDormNumber

### 24. 排班管理是怎么实现自动排班的？

排班管理支持两种模式：
- **按周轮换（weekly）**：每周一个人负责，下周自动轮到下一个人
- **按天轮换（daily）**：每天一个人负责，按成员顺序循环

实现逻辑：
1. 获取宿舍成员列表
2. 根据选择的轮换模式和起始日期，自动计算每个日期对应的值日人员
3. 使用取模运算（`index % memberCount`）实现循环分配
4. 生成的排班记录存入 Schedule 集合，包含周标签、日期、星期、负责人等信息
5. 创建排班后，自动给明天值日的人发送提醒通知

### 25. AI 智能排班是怎么实现的？

这是系统的一个亮点功能，集成了 DeepSeek 大模型 API：
1. 用户用自然语言描述排班需求（如"小明周三有课不排，周末不排班"）
2. 后端获取宿舍成员列表和已有排班数据，构建 system prompt
3. 调用 DeepSeek API（deepseek-chat 模型），将成员信息、日期范围、用户需求作为上下文
4. AI 返回 JSON 格式的排班结果，后端解析并映射 personId
5. 前端展示 AI 生成的排班表，用户确认后保存

关键设计：
- prompt 中明确要求 AI 只返回 JSON 格式，方便解析
- 使用正则提取 JSON（AI 可能返回 markdown 代码块包裹的 JSON）
- 将已有排班数据传给 AI，让它在现有基础上调整而非从零开始
- 设置 temperature=0.3，降低随机性，保证排班结果的稳定性

### 26. 投票决策功能是怎么防止重复投票的？

在 Decision 模型中设计了 `voters` 数组，每个元素记录 `{ userId, optionIndex }`。投票时：
1. 后端先检查投票是否已结束（status !== 'active'）
2. 检查是否已过截止时间（如果过了，自动将状态改为 ended）
3. 检查选项索引是否有效（防止越界）
4. 检查 `voters` 数组中是否已存在当前用户的 userId（`voters.some(v => v.userId.toString() === userId)`）
5. 如果已存在，返回"已投过票"提示
6. 如果未投票，将投票记录 push 到 voters 数组，同时对应选项的 count +1，原子保存

### 27. 消息通知机制是怎么实现的？

系统采用**被动拉取 + 内部推送**的消息通知机制：

**消息生成**：封装了 `createNotification` 工具函数，接收 userIds 数组、消息类型和内容，使用 `Message.insertMany` 批量创建消息记录。各业务模块（费用、投票、排班、公告、宿舍变更等）在完成操作后异步调用此函数。

**消息展示**：
- 前端进入消息页面时请求消息列表，按时间倒序展示
- 支持按类型（vote/task/finance/system）和状态（unread/read）筛选
- 支持关键词模糊搜索（后端使用 MongoDB 的 $regex 正则匹配）
- 未读消息通过 tabBar badge 红点提示（首页 onShow 时查询未读数量）
- 支持单条标记已读和全部标记已读

### 28. 费用管理中的分摊逻辑是怎样的？

费用管理的核心流程：
1. 发起人创建一笔费用记录，填写标题和金额
2. 系统自动将宿舍内其他成员设为付款人（payeeIds 数组）
3. 后端异步通知宿舍其他成员（消息类型为 finance）
4. 费用记录支持"未收款/已收款"状态切换
5. 支持滑动删除操作，方便管理历史记录
6. 列表查询时通过 populate 关联查询创建者和付款人的昵称头像

### 29. 公告功能有什么特别的设计？

公告模块有几个值得一提的设计：
- **已读追踪**：Announce 模型中有 readBy 数组，记录已读用户。用户查看公告详情时，自动将其 userId 加入 readBy（使用 $addToSet 防止重复）
- **已读统计**：列表接口返回 readCount 和 totalMembers，前端可以展示"3/4人已读"
- **创建者自动已读**：创建公告时，创建者自动加入 readBy
- **通知联动**：发布公告后自动给宿舍其他成员发送消息通知

### 30. 宿舍长选举是怎么实现的？

宿舍长选举（实际是转让）的流程：
1. 当前宿舍长选择一个成员作为新宿舍长
2. 后端验证目标用户是否为本宿舍成员
3. 使用 `User.updateMany` 将当前宿舍所有 role 为 leader 的用户降为 member
4. 将目标用户的 role 更新为 leader
5. 异步通知宿舍所有成员"宿舍长已变更为 xxx"

## 五、前端实现细节类

### 31. 国际化（i18n）是怎么实现的？

采用了轻量级的自研 i18n 方案，没有引入第三方库：
- 定义了四个语言包文件（zh-CN、en、ko、ja），每个文件导出一个嵌套的键值对对象
- 利用 Vue 3 的 `ref` 创建响应式的语言标记 `currentLang`
- 翻译函数 `t(key)` 内部访问 `currentLang.value`，使 Vue 自动追踪依赖
- 切换语言时更新 `currentLang`，所有使用 `t()` 的模板自动重新渲染
- 同时调用 `uni.setTabBarItem` 动态更新 tabBar 文字
- 语言偏好存储在本地（uni.setStorageSync），下次打开自动恢复

这种方案的优点是轻量（不到 50 行代码）、无需引入第三方库，且利用了 Vue 3 响应式系统实现自动更新。

### 32. 夜间模式是怎么实现的？

夜间模式通过 **CSS 变量 + 响应式状态** 实现：
1. 在 App.vue 的全局样式中定义两套 CSS 变量（亮色和暗色），如 `--bg-page`、`--text-primary`、`--bg-card` 等
2. 暗色变量通过 `.dark-mode` 类名覆盖
3. 在 `utils/theme.js` 中用 Vue 3 的 `ref` 创建响应式的 `isDark` 状态
4. 每个页面的根元素绑定 `:class="['page', isDark ? 'dark-mode' : '']"`
5. 切换时同步更新导航栏颜色（`uni.setNavigationBarColor`）和 tabBar 样式（`uni.setTabBarStyle`）
6. 主题偏好存储在本地，启动时自动恢复

### 33. 前端的登录态管理是怎么做的？

在 `utils/auth.js` 中封装了完整的认证工具：
- `wxLogin()`：执行微信登录全流程，成功后将 token、userInfo、userId 存入本地
- `isLoggedIn()`：检查本地是否有 token
- `getLocalUserInfo()`：获取本地存储的用户信息
- `clearAuth()`：清除所有登录态（退出登录时调用）

登录态检查时机：
- App.vue 的 onLaunch：启动时检查，未登录停留在登录页，已登录跳转首页
- 首页的 onShow：每次显示时检查，未登录则重定向到登录页
- request.js 的响应拦截：收到 401 时自动清除登录态并跳转

### 34. 头像上传和显示是怎么处理的？

头像处理涉及上传和显示两个环节：

**上传**：
1. 前端调用 `uni.chooseImage` 选择图片
2. 使用 `uni.uploadFile` 将图片上传到后端 `/api/user/upload-avatar`
3. 后端使用 multer 中间件处理文件上传，保存到 uploads 目录，文件名格式为 `avatar_时间戳.扩展名`
4. 返回图片的相对路径（如 `/uploads/avatar_xxx.jpg`）
5. 前端再调用更新用户信息接口，将头像路径保存到用户记录

**显示**：
- 微信小程序真机不支持直接加载 HTTP 图片
- 解决方案：使用 `uni.downloadFile` 将服务器图片下载到本地临时文件，再用临时文件路径显示
- 登录时就执行一次下载，将本地路径存入 userInfo.avatarUrl

### 35. 下拉刷新是怎么实现的？

首页使用 UniApp 的 `scroll-view` 组件实现下拉刷新：
- 设置 `refresher-enabled` 开启刷新功能
- 绑定 `:refresher-triggered="refreshing"` 控制刷新状态
- 监听 `@refresherrefresh` 事件触发刷新逻辑
- 刷新时使用 `Promise.all` 并行请求宿舍信息、待办列表、未读消息数量
- 请求完成后将 refreshing 设为 false，结束刷新动画

### 36. Vue 3 Composition API 相比 Options API 有什么优势？你是怎么用的？

Composition API 的优势：
- **逻辑复用**：可以将相关逻辑抽取为独立的函数（如 auth.js、i18n.js、theme.js），在多个组件中复用
- **代码组织**：相关的状态和方法放在一起，而不是分散在 data、methods、computed 等选项中
- **TypeScript 友好**：更好的类型推导支持
- **更灵活**：可以在 setup 中自由组合响应式状态

在项目中的使用：
- 所有页面使用 `<script setup>` 语法
- 使用 `ref` 定义响应式数据
- 使用 `onShow`、`onLoad` 等生命周期钩子
- 将认证、国际化、主题等逻辑抽取为独立的工具模块

## 六、安全与异常处理类

### 37. 系统在安全方面做了哪些考虑？

1. **身份认证**：使用 JWT 令牌机制，所有业务接口都需要通过 authMiddleware 验证身份
2. **令牌过期处理**：JWT 设置 7 天有效期，过期后返回 401，前端自动清除登录态并引导重新登录
3. **权限控制**：区分宿舍长（leader）和普通成员（member）角色，部分操作（如编辑公告、发起选举）仅宿舍长可执行
4. **输入校验**：Mongoose Schema 层面设置了 required、enum、trim 等校验规则；Controller 层对必填参数进行手动校验
5. **防重复操作**：投票防重复（voters 数组检查）、创建宿舍防重复（检查用户是否已有宿舍、宿舍号是否已存在）、加入宿舍防重复（$addToSet 操作）
6. **跨域控制**：通过 CORS 中间件控制允许的请求来源和方法
7. **错误处理**：全局错误处理中间件捕获未处理的异常，避免敏感信息泄露（如堆栈信息）
8. **环境变量**：敏感配置（数据库连接串、JWT 密钥、微信 AppSecret）通过 .env 文件管理，不硬编码在代码中

### 38. 全局错误处理是怎么实现的？

后端在 `middlewares/errorHandler.js` 中实现了全局错误处理中间件：
- 放在所有路由之后（Express 的错误处理中间件必须有 4 个参数）
- 针对不同错误类型返回不同响应：
  - **ValidationError**（Mongoose 数据验证错误）：提取所有错误信息，返回 400
  - **CastError**（无效的 ObjectId 格式）：返回 400 "无效的ID格式"
  - **其他错误**：返回 500 "服务器内部错误"
- 所有 Controller 方法都使用 try-catch 包裹，catch 中调用 `next(err)` 将错误传递给全局处理

前端也有对应的错误处理：
- request.js 中统一处理网络异常和 HTTP 错误状态码
- 401 自动清除登录态并跳转
- 其他错误弹出 Toast 提示

### 39. 如果 JWT 令牌被盗用了怎么办？

当前的安全措施：
- JWT 有效期设为 7 天，过期后自动失效
- 令牌存储在微信小程序的本地存储中，不同小程序之间隔离

可以进一步改进的方向：
- 引入 Refresh Token 机制，Access Token 设短有效期（如 2 小时），Refresh Token 用于续签
- 在令牌中加入设备指纹，服务端校验请求来源
- 实现令牌黑名单机制（需要 Redis 支持），用户主动退出时将令牌加入黑名单

### 40. CORS 跨域是怎么处理的？为什么需要处理跨域？

**为什么需要**：前后端分离架构下，前端（小程序/H5）和后端（Express 服务）运行在不同的域名/端口上，浏览器的同源策略会阻止跨域请求。

**实现方式**：使用 cors 中间件，配置如下：
- `origin: '*'`：开发阶段允许所有来源（生产环境应限制为具体域名）
- `methods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS']`：允许的 HTTP 方法
- `allowedHeaders: ['Content-Type', 'Authorization']`：允许的请求头（Authorization 用于携带 JWT）
- `maxAge: 86400`：预检请求缓存 24 小时，减少 OPTIONS 请求次数

> 注意：微信小程序原生请求不受浏览器同源策略限制，但 H5 版本和开发调试时需要 CORS 支持。

## 七、性能与优化类

### 41. 如果用户量增大，系统可能会有什么性能瓶颈？你会怎么优化？

可能的瓶颈和优化方向：
- **数据库查询**：随着数据量增长，查询变慢 → 为高频查询字段（dormId、userId、status、createTime）添加索引
- **消息通知**：大量消息写入 → 可引入消息队列（如 Redis）异步处理，避免阻塞主流程
- **文件存储**：头像等文件存在本地 → 迁移到对象存储服务（如阿里云 OSS、腾讯云 COS）
- **接口并发**：单进程 Node.js 的并发上限 → 使用 PM2 进行多进程管理，或引入 Nginx 负载均衡
- **缓存**：热点数据（宿舍信息、成员列表）→ 引入 Redis 缓存减少数据库访问
- **分页**：目前部分列表接口没有分页 → 增加分页参数（page、pageSize），避免一次返回过多数据

### 42. 前端做了哪些性能优化？

- **按需加载**：页面使用 UniApp 的路由懒加载，只在访问时才加载对应页面
- **并行请求**：首页 onShow 时使用 `Promise.all` 并行请求多个接口，减少等待时间
- **本地缓存**：用户信息、语言偏好、主题偏好等存储在本地，避免重复请求
- **下拉刷新**：使用 scroll-view 的原生刷新能力，体验流畅
- **图片处理**：头像下载到本地临时文件，避免每次都从服务器加载

### 43. MongoDB 查询有做过优化吗？

- 使用 `populate` 进行关联查询时，通过第二个参数限制返回字段（如 `populate('creatorId', 'nickname avatar')`），减少数据传输量
- 列表查询使用 `.sort({ createdAt: -1 })` 按时间倒序，配合索引可以高效排序
- 消息搜索使用 `$regex` 正则匹配，支持模糊搜索
- 加入宿舍使用 `$addToSet` 操作，保证成员不重复且是原子操作
- 历史排班查询使用 `.skip(1).limit(10)`，跳过最新一条（当前排班），只返回历史记录

## 十、理论与概念类（老师可能考察基础知识）

### 51. 什么是前后端分离？有什么优缺点？

前后端分离是指前端和后端作为独立的项目开发和部署，通过 API 接口进行数据交互。

**优点**：
- 前后端可以并行开发，提高效率
- 前端可以独立选择技术栈（Vue、React 等）
- 后端 API 可以同时服务多个客户端（小程序、H5、App）
- 职责清晰，易于维护

**缺点**：
- 需要额外处理跨域问题
- 首屏加载可能较慢（需要先加载 JS 再请求数据）
- 需要约定接口规范，增加沟通成本

### 52. 什么是中间件？Express 的中间件机制是怎样的？

中间件是 Express 中处理请求的函数，可以访问请求对象（req）、响应对象（res）和下一个中间件函数（next）。

Express 的中间件机制是**洋葱模型**（实际是线性管道）：请求按照中间件注册的顺序依次经过每个中间件，每个中间件可以：
- 执行任意代码
- 修改 req 和 res 对象
- 结束请求-响应循环（调用 res.json 等）
- 调用 next() 将控制权传递给下一个中间件

在我的项目中使用了以下中间件：
- `cors`：跨域处理
- `express.json()`：解析 JSON 请求体
- `express.static()`：静态文件服务
- `authMiddleware`：JWT 认证（自定义）
- `errorHandler`：全局错误处理（自定义）

### 53. 什么是 RESTful API？

REST（Representational State Transfer）是一种 API 设计风格，核心原则：
- **资源导向**：URL 表示资源（如 /api/user、/api/dorm）
- **HTTP 方法语义化**：GET 查询、POST 创建、PUT 更新、DELETE 删除
- **无状态**：每次请求包含所有必要信息，服务端不存储会话
- **统一接口**：使用标准的 HTTP 状态码和统一的响应格式

### 54. 什么是 JWT？它的结构是怎样的？

JWT（JSON Web Token）是一种用于身份认证的令牌标准，由三部分组成：
- **Header**：声明令牌类型和签名算法（如 HS256）
- **Payload**：载荷，包含用户信息（如 userId、openid）和过期时间
- **Signature**：签名，使用密钥对 Header 和 Payload 进行签名，防止篡改

三部分用 `.` 连接，形如 `xxxxx.yyyyy.zzzzz`。服务端验证时用相同的密钥重新计算签名，与令牌中的签名比对，一致则说明令牌未被篡改。

### 55. MongoDB 和 MySQL 的核心区别是什么？

| 维度 | MongoDB | MySQL |
|------|---------|-------|
| 数据模型 | 文档模型（JSON-like） | 关系模型（表、行、列） |
| Schema | 灵活（Schema-less） | 严格（需预定义表结构） |
| 查询语言 | MongoDB Query Language | SQL |
| 关联 | 嵌入式文档 或 $lookup | JOIN |
| 事务 | 支持（4.0+） | 原生支持 |
| 扩展方式 | 水平扩展（分片） | 垂直扩展为主 |
| 适用场景 | 数据结构多变、快速迭代 | 数据关系复杂、强一致性要求 |

### 56. 什么是 Composition API？和 Options API 有什么区别？

Options API 是 Vue 2 的写法，将组件逻辑按选项分类（data、methods、computed、watch 等）。Composition API 是 Vue 3 引入的新写法，将相关逻辑组织在一起。

主要区别：
- **代码组织**：Options API 按选项分类，Composition API 按功能分组
- **逻辑复用**：Options API 用 mixins（有命名冲突问题），Composition API 用普通函数
- **TypeScript 支持**：Composition API 的类型推导更好
- **灵活性**：Composition API 可以在 setup 中自由组合响应式状态

## 十二、可能的追问与深挖

### 60. Node.js 的事件循环机制是什么？

Node.js 基于事件驱动的非阻塞 I/O 模型。事件循环是 Node.js 处理异步操作的核心机制：
- 主线程执行同步代码
- 遇到异步操作（如数据库查询、文件读写）时，将其交给底层的 libuv 线程池处理
- 异步操作完成后，回调函数被放入事件队列
- 事件循环不断检查队列，取出回调函数执行

这就是为什么 Node.js 单线程也能处理高并发——它不会阻塞在 I/O 操作上。

### 61. async/await 的原理是什么？

async/await 是 Promise 的语法糖：
- `async` 函数返回一个 Promise
- `await` 暂停 async 函数的执行，等待 Promise 解决后继续
- 底层基于 Generator 和 Promise 实现

在我的项目中，所有 Controller 方法都使用 async/await 处理异步操作（数据库查询、微信接口调用等），代码可读性比回调和 .then 链式调用好很多。

### 62. 如果两个用户同时投票，会不会出现数据不一致？

MongoDB 的单文档操作是原子性的。在我的投票实现中：
1. 先读取 Decision 文档
2. 检查是否已投票
3. 修改 options 和 voters
4. 调用 save() 保存

理论上在高并发场景下，两个请求可能同时读取到"未投票"状态，导致重复投票。解决方案：
- 可以使用 MongoDB 的 `findOneAndUpdate` 配合 `$push` 和条件查询，将读取和写入合并为一个原子操作
- 或者使用 MongoDB 的乐观锁（version key）机制
- 当前项目的用户量（一个宿舍 4-6 人）不太可能出现这种并发问题

### 63. 为什么消息通知用异步方式发送？

在各个 Controller 中，消息通知都是在主要操作完成并返回响应之后，通过 try-catch 包裹的异步调用发送的。这样设计的原因：
- **不阻塞主流程**：即使通知发送失败，也不影响主要操作的成功响应
- **提升响应速度**：用户不需要等待通知发送完成
- **容错性**：通知失败只打印错误日志，不会导致整个请求失败

### 64. populate 的原理是什么？和 SQL 的 JOIN 有什么区别？

Mongoose 的 populate 本质上是**额外查询**：
1. 先查询主文档（如 Finance）
2. 收集所有需要关联的 ObjectId
3. 发起第二次查询，根据这些 ID 查询关联文档（如 User）
4. 将查询结果填充到主文档的对应字段中

与 SQL JOIN 的区别：
- JOIN 是在数据库层面完成的，一次查询返回结果
- populate 是在应用层面完成的，需要多次查询
- populate 的性能不如 JOIN，但 MongoDB 的文档模型通常通过嵌入式设计减少关联查询的需求

### 65. 你的系统支持多少并发用户？

当前架构下的并发能力分析：
- Node.js 单进程理论上可以处理数千个并发连接（得益于非阻塞 I/O）
- MongoDB Atlas 免费套餐支持 500 个并发连接
- 实际瓶颈取决于数据库查询复杂度和网络延迟

对于宿舍管理场景（每个宿舍 4-6 人，同时在线用户不会太多），当前架构完全够用。如果需要扩展，可以使用 PM2 多进程、Nginx 负载均衡、Redis 缓存等方案。
