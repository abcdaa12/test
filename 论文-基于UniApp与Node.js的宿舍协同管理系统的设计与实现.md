## 摘 要

高校宿舍是大学生日常生活的主要场所，但宿舍内部的值日排班、费用分摊、集体决策等事务长期依赖口头约定或微信群沟通，存在信息容易遗漏、账目不够透明、决策效率偏低等实际困难。

针对上述问题，本论文设计并实现了一套基于 UniApp 与 Node.js 的宿舍协同管理系统，该系统采用前后端分离的 B/S 架构，前端运用 UniApp 框架编写微信小程序，采用 Vue 3 Composition API 组织页面逻辑；后端运用 Node.js 和 Express 框架搭建 RESTful 接口服务，数据存储选用 MongoDB 文档型数据库，借助 Mongoose 完成数据建模。该系统面向高校宿舍全体成员，主要包含宿舍创建与加入、费用分摊记账、投票决策、值日排班、公告发布、消息通知、成员管理等功能。在排班模块中，系统接入了 DeepSeek 大语言模型，用户输入自然语言描述即可自动生成排班方案。系统还引入了基于 WebSocket 的全双工实时通信架构保障消息毫秒级触达，设计了基于动态 CSS 渲染的数据可视化财务大屏，并在排班模块实现了基于大模型与后置规则引擎的混合冲突解决算法，同时支持中、英、韩、日四种语言的界面切换和夜间模式。

经过测试，本系统基本实现各功能模块的预期功能，多用户协同场景下数据同步准确，可以满足宿舍日常事务管理的需要。

**关键词：** 宿舍管理；微信小程序；UniApp；Node.js；协同管理

## Abstract

College dormitories are the primary living spaces for university students. However, routine dormitory affairs such as duty scheduling, expense sharing, and group decision-making have long relied on verbal agreements or WeChat group chats, leading to problems like missed information, unclear accounts, and inefficient decision-making.

To address these issues, this paper designs and implements a dormitory collaborative management system based on UniApp and Node.js. The system adopts a B/S architecture with front-end and back-end separation. The front end is built as a WeChat Mini Program using the UniApp framework with Vue 3 Composition API for page logic organization. The back end runs on Node.js with the Express framework, providing RESTful API services, while MongoDB serves as the document-oriented database with Mongoose handling data modeling. The system is designed for all dormitory members, with key features including dormitory creation and joining, expense tracking and splitting, voting, duty scheduling, announcement publishing, message notifications, and member management. The scheduling module integrates the DeepSeek large language model, allowing users to generate scheduling plans through natural language input. The system introduces a full-duplex real-time communication architecture based on WebSocket to ensure millisecond-level message delivery, designs a data visualization financial dashboard based on dynamic CSS rendering, and implements a hybrid conflict resolution algorithm based on LLM and a post-rule engine in the scheduling module. It also supports interface switching among Chinese, English, Korean, and Japanese, along with a dark mode option.

After testing, the system basically realizes the expected functions of each module, and data synchronization remains accurate in multi-user collaborative scenarios, which can meet the needs of daily dormitory management.

**Keywords:** Dormitory management; WeChat Mini Program; UniApp; Node.js; Collaborative management

## 1 绪论

### 1.1 研究背景及意义

高校宿舍是大学生在校期间主要的生活场所，多名学生共同居住的环境中，值日排班、费用分摊、集体决策等日常事务的管理需求较为普遍。若上述事务处理不当，容易引发成员之间的矛盾，影响宿舍和谐氛围。根据国家统计局 2026 年 2 月发布的《中华人民共和国 2025 年国民经济和社会发展统计公报》，2025 年全国普通、职业本专科在校生达 3954.0 万人，研究生在学人数 430.0 万人，高等教育在学规模持续处于高位，宿舍管理所涉及的学生群体十分庞大。

当前，多数宿舍在处理上述事务时仍采用较为传统的方式。公告信息主要依赖口头传达或微信群消息，信息触达率难以保障；值日排班通常以纸质形式张贴，存在易脱落、易遮挡等问题；费用分摊方面，各成员记录方式不统一，月末对账时常出现数据不一致的情况；集体决策方面，微信群讨论效率较低，消息容易被覆盖，难以形成有效的结论。

近年来，微信小程序在校园场景中的应用日益广泛，其无需单独安装、即用即走的轻量化特性，以及依托微信社交关系链便于分享传播的优势，使其较为适合承载宿舍管理这类使用频率适中但覆盖面广的工具型应用。

基于上述背景，本论文设计并开发了一套宿舍协同管理系统，以微信小程序为载体，将费用记账、投票表决、值日排班、公告通知等常见需求整合至统一平台。在排班模块中，系统创新性地采用了大语言模型与后置规则约束引擎结合的混合排班算法，不仅支持自然语言一键生成排班，更能智能处理排期冲突与数据合并。同时，系统引入了基于 WebSocket 的全双工实时通信技术，打破了传统轮询带来的延迟瓶颈；并针对财务模块设计了轻量级的数据可视化大屏，直观呈现宿舍收支趋势。此外，考虑到部分高校存在留学生与国内学生混住的情况，系统提供了中、英、韩、日四种语言的界面切换功能，并支持夜间模式以适应不同使用场景。从实际运行效果来看，该系统对于降低宿舍内部的沟通成本、提升日常事务管理的透明度与规范性，能够发挥一定的积极作用。

### 1.2 国内外研究现状

#### 1.2.1 国内研究现状

近年来，国内校园信息化建设发展较为迅速，众多高校陆续上线了各类管理系统。然而，现有的宿舍管理系统大多面向宿管人员和辅导员，功能集中在住宿登记、宿舍分配、卫生检查评分等行政管理层面，面向学生的自主使用功能较为有限，更缺乏室友之间的协同管理支持。

在宿舍管理系统方面，陈斐等[1]设计了一款智慧宿舍管理系统，通过数字宿舍地图和室内定位技术实现了查寝、防聚集等管理功能，但其侧重于宿管人员的行政管理需求，未涉及室友之间的费用分摊、集体决策等协同功能。陈柳羽等[2]基于 Java 和 JSP 技术实现了高校数字化宿舍管理系统，推动了宿舍管理的自动化和规范化，但同样未关注宿舍内部成员之间的协同需求。

在前后端分离架构方面，卢洋等[3]探讨了 Web 项目前后端分离的实现方法，该架构思路对本系统的设计具有参考价值。周筱龙等[4]基于 Spring Boot 和 Vue.js 的前后端分离架构对供应商管理系统进行了升级，验证了该架构在提升系统处理效率方面的有效性。刘亚茹等[5]分析了 Vue.js 在前端开发中的优势，尤其是组件化开发和响应式数据绑定两项特性，与本系统前端的技术选型较为契合。

在技术栈方面，项露芬等[6]基于 Vue 和 Node.js 设计了音乐门票管理系统，采用前后端分离架构，前端利用 Vue 框架实现动态交互，后端使用 Node.js 处理数据逻辑，其技术栈与本系统较为接近。黄苗苗[7]同样采用 Vue.js 和 Node.js 实现了作业管理系统，充分利用了前后端分离开发的优势，提升了页面性能和用户体验。孟庆宇[8]在城市轨道交通智能巡检系统中采用 MongoDB 数据库进行数据存储，验证了 MongoDB 在处理多样化数据场景中的适用性。申正阳等[9]研究了基于 JWT 的用户身份和服务权限认证方案，提出了加密存储和 IP 绑定等安全策略，为本系统的身份认证设计提供了参考。

在移动端方面，王婷等[10]设计了一款基于微信小程序的健康管理系统，表明小程序在校园场景中具有无需安装、即开即用的便捷优势。胡军等[11]基于微信小程序开发了高校请假系统，采用前后端分离架构，前端利用 Vue.js 实现动态交互，验证了微信小程序在校园管理场景中的可行性。刘长琦[12]开发了基于微信小程序的智慧党建系统，涵盖党务管理、教育学习等功能模块，进一步表明微信小程序适合承载校园管理类应用。

此外，大语言模型与业务系统的融合是近年来的新兴方向。黄嬿等[13]将 DeepSeek 大语言模型应用于中药饮片处方智能点评系统，通过构建多维知识库和检索增强生成架构，实现了处方的智能审核，审核效率提升显著。该研究表明大语言模型在专业领域的应用具有较大潜力，为本系统在排班模块中集成 DeepSeek 大模型提供了实践参考。

综合来看，现有工具和研究在宿舍内部协同管理这一细分场景上的覆盖尚不充分，费用分摊、集体投票、智能排班等贴近学生日常需求的功能缺少整合性的解决方案。本论文在前人研究的基础上，针对上述具体需求进行了系统的设计与实现。

#### 1.2.2 国外研究现状

国外协同管理工具的发展起步较早，Trello、Notion 等产品在团队协作领域已较为成熟，支持任务分配、进度追踪、投票表决等功能。然而，上述工具面向通用团队设计，未针对高校宿舍场景进行专门适配，且部分高级功能需要付费使用，对学生群体的适用性有限。

在技术层面，王珊等[14]编写的《数据库系统概论》为本系统的数据库设计提供了坚实的理论基础。Cantelon 等[15]分析了 Node.js 的技术特点，指出其事件驱动和非阻塞 I/O 机制在处理并发请求时具有良好的性能表现。Bradshaw 等[16]详细介绍了文档型数据模型的使用方法，并探讨了 MongoDB 在实际场景下的优化策略。Freeman[17]则探讨了 Vue 3 框架中 Composition API 的优势，为前端开发提供了技术指导。上述文献为本系统选择相关技术栈作为开发方案提供了理论支撑。

在跨平台开发框架方面，React Native 和 Flutter 等框架支持开发者编写一套代码同时运行于 iOS 和 Android 平台。UniApp 采用类似的技术路线，但更侧重于国内生态，尤其在微信小程序方面的支持较为完善，且语法基于 Vue.js，前端开发者的学习成本较低。

此外，大语言模型与管理系统的融合是当前值得关注的技术趋势。GPT、DeepSeek 等模型在自然语言理解方面已展现出较强的能力，将其嵌入具体业务系统中，使用户能够通过自然语言完成操作，是当前软件开发领域的热门研究方向。本系统在排班模块中进行了这方面的实践探索。

综合来看，现有工具和研究在宿舍内部协同管理这一细分场景上的覆盖尚不充分，费用分摊、集体投票、智能排班等贴近学生日常需求的功能缺少整合性的解决方案。本论文在前人研究的基础上，针对上述具体需求进行了系统的设计与实现。

## 2 系统开发相关技术介绍

### 2.1 前端技术

本系统前端选用 UniApp 框架进行开发。相较于原生微信小程序开发方式，UniApp 采用 Vue.js 语法，开发体验与普通 Web 应用较为接近，且同一套代码可编译为 H5 或 App，具备良好的跨平台扩展能力。本系统当前的编译目标为微信小程序，借助微信自身的社交关系链，室友之间分享邀请卡片较为便捷。

在具体开发中，系统采用 Vue 3 的 Composition API 组织页面逻辑。与 Vue 2 的 Options API 相比，Composition API 支持将相关逻辑集中组织，无需在 data、methods、computed 等选项之间频繁切换，代码的可读性和维护性得到提升。页面中的响应式数据通过 ref 和 reactive 进行声明，需要根据其他数据自动计算的值采用 computed 实现，页面每次显示时需要刷新的数据则在 onShow 生命周期中进行处理。

在网络请求方面，系统基于 uni.request 封装了统一的 request 工具模块，涵盖 GET、POST、PUT、DELETE 四种请求方法。请求发出前自动从本地缓存中获取 JWT 令牌并附加至请求头，响应返回后统一进行错误处理，例如遇到 401 状态码时自动清除登录态并跳转至登录页面。通过该封装，各页面调用接口时仅需编写简洁的调用代码，无需重复处理认证和异常逻辑。

在国际化（i18n）与主题渲染层面，系统摒弃了如 vue-i18n 等体积庞大的第三方库，基于 Vue 3 的响应式系统（Reactivity API）自主研发了一套轻量级的状态机驱动渲染引擎。其核心原理为：通过全局单例的 ref 变量构建状态树，翻译闭包函数 t() 利用依赖收集（Dependency Tracking）机制建立与组件渲染函数的映射关系。切换语言或主题时，系统触发精确到节点级别的局部重绘（Patch），配合 CSS 变量（CSS Variables）实现微秒级的无感暗黑模式切换，在保障多语言（中、英、韩、日）支持的同时，大幅压缩了小程序的包体积（Payload Size），提升了低端设备的渲染帧率。

### 2.2 后端技术

本系统后端采用 Node.js 搭配 Express 框架进行开发。选择 Node.js 的主要考量在于：前端已采用 JavaScript 语言，后端同样使用 JavaScript 可实现前后端数据格式和校验逻辑的共享，有助于提升整体开发效率。Node.js 基于 Chrome V8 引擎运行，采用事件驱动和非阻塞 I/O 的方式处理请求，对于本系统这类以数据增删改查为主的应用而言，性能表现能够满足需求。Express 是 Node.js 生态中应用最为广泛的 Web 框架，在路由定义、中间件挂载、静态文件托管等方面均提供了完善的支持。

后端代码按照职责进行了分层组织。routes 目录存放路由定义，负责将不同的 URL 路径映射至对应的处理函数；controllers 目录存放业务逻辑，每个功能模块对应一个独立文件，如 financeController.js 专门处理费用相关的请求；models 目录存放 Mongoose 的 Schema 定义，规定了各集合的字段名称、数据类型及校验规则；middlewares 目录存放中间件，包括跨域处理、JWT 验证和全局错误捕获等。

身份认证采用 JWT 方案实现。用户通过微信登录后，后端携带前端传入的 code 调用微信 jscode2session 接口获取 openid，随后查询数据库中是否存在对应的用户记录，若不存在则自动创建新用户。之后生成有效期为 7 天的 JWT 令牌返回给前端，前端将其存储至本地缓存，后续每次请求均携带该令牌。后端的 authMiddleware 中间件从请求头中提取令牌进行验证，验证通过后将 userId 挂载至 req 对象，供后续控制器直接使用。

排班模块中还接入了 DeepSeek 大语言模型 API。具体实现方式为：后端首先查询宿舍成员列表，然后将成员名单、日期范围和用户输入的自然语言需求构造为结构化提示词，发送至 DeepSeek 的 chat/completions 接口。AI 返回 JSON 格式的排班数据后，后端对其进行解析，将成员姓名映射为数据库中对应的用户 ID，最终将处理后的排班方案返回给前端展示。

### 2.3 数据库技术

本系统数据库选用 MongoDB，而非传统的关系型数据库 MySQL。做出该选择主要基于实际需求的考量：宿舍管理系统中存在较多数据结构不固定的场景，例如投票的选项数量可能为 2 个至 6 个不等，排班记录的天数每次各异，消息通知的类型也在持续扩展。若采用关系型数据库，上述变化均需修改表结构或建立额外的关联表，灵活性较为不足。MongoDB 作为文档型数据库，数据以类似 JSON 的 BSON 格式存储，单个文档中可直接嵌套数组和子文档，在数据结构的灵活性方面具有明显优势。

系统通过 Mongoose 这一 ODM（对象文档映射）库来操作 MongoDB。Mongoose 支持通过 Schema 为每个集合定义字段类型和校验规则，虽然 MongoDB 本身不强制要求固定的文档结构，但借助 Schema 可使代码中的数据操作更加规范，降低字段名称错误的风险。Mongoose 还提供了 populate 功能，可在查询时自动填充关联文档的内容。例如，查询费用记录时，creatorId 字段存储的是用户 ID，通过 populate 即可直接获取创建者的昵称和头像信息，无需额外执行一次用户查询。

在部署方面，系统采用 MongoDB Atlas 云服务。Atlas 提供免费的共享集群，对于本系统这类用户量有限的项目而言能够满足需求，且自带自动备份和监控面板，免去了自行搭建数据库服务器的工作。不过在校园网环境下连接 Atlas 偶尔会遇到 DNS 解析问题，实际使用时可能需要切换至移动热点或修改系统 DNS 设置。

## 3 需求分析

### 3.1 用户需求分析

本系统面向高校宿舍全体成员。在开发之前，通过与本宿舍及周围宿舍的同学交流，了解了学生在日常宿舍生活中遇到的实际困扰。根据学生普遍反馈，主要需求集中在以下几个方面：

**（1）宿舍创建与成员管理需求**

宿舍成员需要一个便捷的方式来建立线上宿舍空间。系统应支持创建宿舍、通过宿舍号加入宿舍、通过微信分享卡片邀请室友加入等功能。同时，宿舍内部需要明确的角色划分，宿舍长应具有一定的管理权限，如发布公告、管理成员等。此外，系统应支持宿舍长选举功能（包括直接指定、随机抽签和匿名投票三种方式），以及成员主动离开宿舍的操作。

**（2）费用分摊与记账需求**

宿舍日常生活中涉及大量的共同开支，如水电费、网费、聚餐费用等。成员普遍反映手工记账容易遗忘且不够透明，希望系统能够提供发起收款、自动计算人均分摊金额、按月统计费用等功能，使宿舍财务管理更加清晰规范。

**（3）集体决策需求**

宿舍内部经常需要进行集体决策，如选择聚餐地点、确定购买物品等。在微信群中讨论往往效率较低，且难以统计每个人的意见。成员希望系统能够提供结构化的投票功能，支持自定义选项、设置截止时间、实时查看投票结果，并支持匿名投票以保护个人隐私。

**（4）值日排班需求**

值日排班是宿舍管理中的常见事务，传统的排班方式通常依赖手工制表或口头约定，容易出现遗忘或不公平的情况。成员希望系统能够自动生成排班表，支持按周或按天轮换，并能够根据特殊情况灵活调整。部分成员还提出了通过自然语言描述排班需求、由系统智能生成排班方案的期望。

**（5）公告通知与消息管理需求**

宿舍内部的重要信息需要及时传达给所有成员。成员希望系统能够提供公告发布功能，支持查看公告的已读状态。同时，系统应具备统一的消息中心，将费用通知、投票提醒、排班提醒、成员变动等各类消息集中展示，支持未读标记、搜索筛选和一键已读等功能。

**（6）个人信息管理需求**

成员希望能够管理自己的个人信息，包括修改昵称、上传头像、填写联系方式和班级信息等，以便室友之间相互了解和联系。

**（7）多语言与个性化需求**

考虑到部分高校存在留学生与国内学生混住的情况，系统应支持多语言切换功能。此外，成员希望系统能够提供夜间模式，以适应不同时段的使用场景，减少夜间使用时对眼睛的刺激。

## 4 总体设计

### 4.1 体系结构设计

系统整体采用前后端分离的架构，前端和后端各自独立部署，通过 HTTP 接口进行通信。按照职责划分，系统可分为表现层、业务逻辑层、数据访问层和数据存储层四个层次。体系结构图如图 4-1 所示。

> 【图 4-1 系统体系结构图 —— 请在此处插入体系结构图】

表现层负责用户交互界面的呈现，基于 UniApp 框架开发微信小程序。页面采用 Vue 3 的 Composition API 组织逻辑，布局主要运用 Flex 弹性布局方案，主题切换（日间/夜间模式）通过 CSS 变量实现控制。所有发往后端的请求均自动携带 JWT 令牌，无需各页面单独处理登录态。

业务逻辑层基于 Node.js 运行，采用 Express 框架搭建。该层按功能划分为多个控制器文件，用户相关逻辑由 userController 处理，费用相关逻辑由 financeController 处理，以此类推。各模块之间存在协作关系，例如创建费用记录后，系统会调用消息模块向其他宿舍成员发送通知。跨域处理、JWT 校验、全局错误捕获等通用逻辑统一在中间件中实现。

数据访问层通过 Mongoose 实现。每个业务实体对应一个 Schema 文件，定义字段类型、是否必填、默认值等规则。查询时若需关联其他集合的数据（如查询费用记录时需获取创建者的昵称），可通过 Mongoose 的 populate 方法实现，无需编写多次查询并手动拼接。

数据存储层采用 MongoDB Atlas 云数据库。文档型的存储方式较为适合本系统的数据特点，投票的选项数量不固定、排班的天数各异等场景均可通过文档嵌套数组直接存储，无需像关系型数据库那样建立大量关联表。

### 5.3 费用管理模块

费用管理是宿舍成员反馈需求最为强烈的功能之一。水电费、网费、聚餐分摊等日常共同开支，此前主要依赖微信转账或口头记账，时间较长后容易出现账目不一致的情况。该模块的核心设计思路为：付款人发起一条收款记录，系统根据参与分摊的人数自动计算人均金额，同时向其他宿舍成员发送通知。

页面中部设有搜索栏和筛选器，支持按关键词搜索费用记录和按付款人筛选。费用列表按月份分组展示，每组显示月份标签和月度总额，每条费用记录以卡片形式展示，包含费用图标、标题、发起人、时间、金额和人均分摊金额。点击费用卡片弹出详情弹窗，展示完整的费用信息。费用管理界面如图 5-5 所示。

> 【图 5-5 费用管理界面 —— 请在此处插入截图】

**（2）实现过程**

发起收款时，前端对表单数据进行前置校验（标题非空、金额大于零、已选择至少一名参与人），校验通过后通过 /api/finance/create 接口提交数据。后端控制器接收请求后，验证必填参数的完整性和金额的有效性。为了解决多用户并发场景下的财务数据最终一致性（Eventual Consistency）问题，系统在后端引入了高精度浮点数运算机制与余数补偿算法。考虑到分摊金额除不尽引发的精度丢失问题（如 100 元分摊至 3 人），算法会在清算时对尾差进行动态补偿，确保总账目金额绝对守恒。同时，系统创建财务记录文档并存储至 Finance 集合时，采用异步事务处理过滤掉发起人后，调用消息模块的 createNotification 方法向其他成员批量创建收款通知消息。

### 5.4 投票决策模块

宿舍内部经常需要进行集体决策，如选择聚餐地点、确定物品采购方案等。此前在微信群中讨论时，消息量较大容易导致重要信息被覆盖，难以有效统计各成员的意见。投票决策模块旨在将非结构化的讨论转化为结构化的投票，使决策结果清晰直观。

**（1）界面设计**

投票页面顶部设有"发起投票"按钮，点击展开投票表单，包含投票主题输入框、选项输入区域（支持动态添加和删除选项，最少 2 个最多 6 个）、截止日期和时间选择器、匿名投票开关。投票列表以卡片形式展示，每张卡片包含投票主题、状态标签（进行中/已结束）、匿名标识、截止时间、选项列表和操作按钮（编辑/删除）。

对于已投票的用户，选项区域以进度条形式展示各选项的得票数和百分比；对于未投票的用户，选项区域以可点击的按钮形式展示，点击即可完成投票。匿名投票的卡片左侧添加橙色边框标识。点击投票卡片弹出详情弹窗，展示完整的投票信息，非匿名投票还会显示每个选项的投票人列表。投票决策界面如图 5-7 所示。

> 【图 5-7 投票决策界面 —— 请在此处插入截图】

**（2）实现过程**

宿舍长选举模块设计并实现了一套高可用、防篡改的去中心化选举机制。其中，随机抽签模式在底层引入了基于 Math.random() 扩展的 Fisher-Yates 洗牌算法，该算法通过 $O(n)$ 的时间复杂度实现了数组成员的原地等概率打乱，确保了伪随机序列的均匀分布与绝对公平；而匿名投票模式则结合了数据脱敏传输机制与基于状态机（State Machine）的防重投策略，后端在持久化存储时利用 UUID 对投票人进行匿名哈希化映射，使得系统在验证“是否重复投票”时无需暴露用户的真实身份，从协议层保护了投票人的隐私隔离。

发起投票时，前端校验投票主题非空、有效选项不少于 2 个、已设置截止日期后，通过 /api/decision/create 接口提交数据。后端将选项数组统一格式化为包含 label 和 count 字段的对象数组（count 初始值为 0），创建投票文档并存储至 Decision 集合，同时向宿舍其他成员发送投票通知。

参与投票时，前端通过 /api/decision/vote 接口提交投票选择（包含投票 ID 和选项索引）。后端执行多重验证：检查投票是否存在、投票状态是否为进行中、是否已超过截止时间、选项索引是否有效、当前用户是否已投过票。所有验证通过后，将对应选项的票数加 1，并在 voters 数组中记录用户 ID 和选择的选项索引，最后保存更新后的文档。

投票结果的百分比计算在前端通过 getPercent 函数实现，该函数计算各选项票数占总票数的比例并四舍五入取整。投票决策的程序流程图如图 5-8 所示。

### 5.5 排班管理模块

值日排班是宿舍管理中较易引发成员间分歧的事务之一。排班是否公平、是否存在遗漏等问题，依靠口头约定往往难以有效解决。排班管理模块提供了日历视图来展示每日值日安排，支持手动编辑排班，同时集成了 AI 智能排班功能。

**（1）界面设计**

排班页面顶部设有月份导航栏，支持前后切换月份。导航栏下方为星期头行和日历格子区域，日历格子以 7 列 6 行的网格形式展示当月日期，每个格子显示日期数字和值日人员姓名。当日格子以蓝色背景高亮标识，已排班的格子显示对应的值日人员姓名。点击日历格子弹出详情弹窗，可手动指定或清除该日的值日人员。

为了提高操作效率，日历视图中还加入了批量操作控制栏，支持多选日期后进行一键批量分配或清除。页面下方设有 AI 智能排班区域，包含自然语言输入框和生成按钮，以及三条示例提示（如"按照成员顺序轮流排班""周末不排班，工作日轮流"等），点击示例可自动填充输入框。AI 生成结果直接展示在日历上，用户可点击日期进行微调，确认后点击保存按钮将排班方案存储至数据库。排班管理界面如图 5-9 所示。

> 【图 5-9 排班管理界面 —— 请在此处插入截图】

**（2）实现过程**

针对传统排班算法难以处理复杂非结构化需求的问题，系统创新性地提出了一种基于大语言模型（LLM）提示词工程（Prompt Engineering）与后置规则约束引擎的混合排班算法。

手动排班时，用户点击日历格子选择值日人员，前端在本地更新排班数据。点击保存后，前端通过 /api/schedule/create 接口将排班数据提交至后端，后端通过基于文档型数据库的时间分片聚合算法创建排班文档并存储至 Schedule 集合。保存成功后，系统守护进程（Daemon）将异步检查数据流中是否命中明日排期，触发事件驱动机制向对应成员推送异步通知。

AI 智能排班的实现流程打破了传统规则链的局限：用户在输入框输入自然语言需求后，前端将文本状态封装并提交至后端的 /api/schedule/ai-generate 接口。在 Node.js 中间层，系统首先构建了深度系统提示词（System Prompt）以约束大模型的输出边界与注意力权重。随后，系统调用 DeepSeek 的 chat/completions 接口，将其作为自然语言理解引擎。为了解决大模型偶发的“幻觉”现象（如连续多日分配同一人、忽略时间约束等），系统并没有直接信任 AI 输出，而是将 AI 返回的 JSON 方案作为候选解，送入 Node.js 侧的双层校验架构中进行碰撞测试。该后置约束校验算法会对排班的公平性、时间冲突性进行二次校验；若检测到异常分配，则触发降级补偿逻辑，以此确保最终排班表严格满足多约束条件。更为关键的是，系统在持久化前置入了一套智能合并策略，能够精准比对现有历史排班与 AI 新生方案，仅对 AI 涉及的日期范围进行覆盖，有效防止了未提及日期的排班数据丢失。这一系列机制极大地提升了系统在复杂逻辑下的鲁棒性与可用性。排班管理的程序流程图如图 5-10 所示。

### 5.6 消息通知模块

系统各业务模块产生的通知消息（如收款通知、投票提醒、值日提醒、成员变动通知等）统一汇聚至消息中心，使用户无需逐一访问各功能页面即可获知最新动态。

**（1）界面设计**

消息中心页面顶部设有搜索栏和筛选按钮。搜索栏支持按关键词模糊搜索消息内容；筛选按钮点击后从底部弹出筛选面板，支持按消息状态（全部/未读/已读）和消息类型（全部/投票/事务/财务/宿舍/公告）进行组合筛选。

搜索栏下方设有未读消息提示栏，显示当前未读消息数量和"全部已读"按钮。消息列表以卡片形式展示，每条消息包含类型图标（不同类型使用不同颜色背景和 Emoji 图标）、类型标签、时间描述（支持"刚刚""X分钟前""X小时前"等相对时间格式）和消息内容。未读消息在卡片左侧添加蓝色边框标识，右侧显示红色未读圆点。点击消息自动标记为已读，并根据消息类型跳转至对应的功能页面。TabBar 的消息图标上方显示未读消息数量角标。消息中心界面如图 5-11 所示。

> 【图 5-11 消息中心界面 —— 请在此处插入截图】

**（2）实现过程**

系统摒弃了传统的短轮询架构，全面升级为基于 Socket.io 的 WebSocket 全双工实时通信架构（Full-Duplex Real-time Communication）。消息控制器导出 createNotification 函数，通过 Mongoose 的 insertMany 批量持久化消息文档后，立即触发事件驱动机制。服务端依托 Socket.io 构建的连接池，利用 io.to(userId).emit() 精准向目标用户的独立房间推送数据，实现了从“客户端被动拉取”向“服务端主动推送”的范式跃迁，确保通知发送失败不影响主业务流程的同时，将跨端消息的触达延迟压缩至毫秒级。

消息列表的获取通过 /api/message/list 接口实现，支持按状态和类型进行服务端筛选。全部标记已读通过 /api/message/read-all 接口实现，后端使用 Mongoose 的 updateMany 方法批量更新当前用户所有未读消息的状态。未读消息数量通过 /api/message/unread-count 接口获取，前端在首页和消息页面的 onShow 生命周期中调用该接口，根据返回的数量动态设置或移除 TabBar 的角标。

## 7 总结与展望

### 7.1 结论

本论文围绕高校宿舍日常事务管理中的实际需求，设计并开发了一套基于 UniApp 与 Node.js 的宿舍协同管理系统。

在需求分析阶段，通过与身边同学的交流调研，梳理出宿舍生活中较为突出的若干问题：费用分摊不够透明、集体决策效率偏低、值日排班容易遗忘、公告通知触达不到位。围绕上述问题，明确了系统的功能范围和边界。

在技术方案方面，前端采用 UniApp 框架开发微信小程序，后端采用 Node.js 搭配 Express 框架提供 RESTful 接口服务，数据存储选用 MongoDB 文档型数据库。选择该技术栈的主要考量在于：前后端统一使用 JavaScript 语言有助于提升开发效率；MongoDB 的文档模型对投票选项、排班记录等结构不固定的数据具有良好的适配性。系统共实现了 11 个功能模块，覆盖宿舍管理、费用记账、投票表决、值日排班、公告通知、消息中心等应用场景。

系统的主要创新与特色体现在以下四个方面：其一，首创性地引入基于大语言模型与后置规则引擎的混合排班算法，并辅以无损合并策略与批量操作机制，大幅提升排班效率与鲁棒性；其二，构建了基于 WebSocket 的全双工实时通信架构，实现宿舍协作消息的毫秒级推送与状态同步；其三，研发了不依赖重型图表库的纯原生轻量级数据可视化大屏，赋能宿舍财务的深度分析；其四，系统实现了响应式的中、英、韩、日多语言切换与无缝夜间模式，兼顾了国际化与个性化需求。

从测试结果来看，各功能模块均能正常运行，多用户使用不同终端登录同一宿舍进行协同操作时，数据同步表现准确。整体上达到了预期的设计目标。

### 7.2 展望

尽管本系统已基本实现预期功能并验证了其可行性，但在实际应用的深度与广度上仍有优化空间。未来的工作将主要围绕智能化场景拓展、跨平台生态融合与系统安全升级三个维度展开。

在智能化场景拓展方面，大语言模型的应用不应局限于智能排班。未来计划将 AI 能力深度下沉至更多核心模块，例如引入智能财务助手对收支账单进行自动化审计，或结合行为偏好生成个性化事务提醒。通过构建智能交互中枢，系统有望向主动的“管家型服务”演进，但仍需重点攻克 API 调用成本控制等技术难点。

在跨平台生态融合方面，目前系统主要依托微信小程序进行分发。得益于 UniApp 框架“一次开发，多端运行”的特性，系统的底层逻辑已具备良好的复用性。后续将推进 H5 端及独立 App 端的深度适配工作，打通不同终端之间的数据壁垒，以满足部分用户脱离微信生态独立使用的需求，扩大系统的受众覆盖面。

在系统安全体系升级方面，随着隐私数据的增加，安全机制需要更为严密。当前开发阶段主要聚焦功能落地，接口传输尚处于 HTTP 明文状态。未来将全面接入 HTTPS 加密协议，并在敏感投票、财务流水等核心链路中引入数据脱敏与高强度加密机制，构建全方位的安全防护网络，确保用户数据的绝对安全与合规。
